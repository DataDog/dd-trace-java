# Performance Review Checks ‚Äî AI-review cost model

Operationalizes the narrative **Performance Review Guide** (`guide.md`) into diff-applicable checks for AI PR review. Complements benchmark-based regression-blocking: benchmarks catch *measured* regressions on *covered* ops; this catches *un-benchmarked* pattern-smells in *any* diff. Grounded in the guide's **Tracer Principles** (do-no-harm, assume-hot) and its severity model. Read this alongside `guide.md` ‚Äî the guide is the narrative "how"; this is the precise confidence/severity cost-model.

## Posture (read first)
- **Advisory, not blocking** initially. A nondeterministic, false-positive-prone check that blocks merges dies of being ignored (false positives) or gives false confidence (false negatives). Earn blocking only after precision is proven, and likely only on the deterministic-lint subset.
- **Precision over recall ‚Äî silent when unsure.** Only flag high-confidence issues on real hot paths. Over-flagging kills the check. (This fights the model's default to be comprehensive/helpful ‚Äî state it explicitly in the prompt.)
- **Resolve-via-sink before flagging callbacks and hooks.** A per-span callback or scope hook registration *looks* like a hot-path cost but may be safe once you follow into the registered listener. If every reachable sink is an atomic counter (`LongAdder`, `AtomicLong`) or no-op-when-disabled, stay silent ‚Äî a "verify contention" nudge there is noise. Follow the full listener chain before emitting a finding.
- **Findings are prompts to *verify*, not verdicts.** The AI reasons statically ‚Äî by the doc's own "Don't Assume; Measure," it *cannot* render a perf verdict. Every finding routes into Benchmark ‚Üí Profile ‚Üí Improve ‚Üí Guard. Phrase as "this looks like X; verify with Y."
- **Triage by severity** (from the doc): flag SEV-1 (memory/OOM) patterns *aggressively* (a false positive is worth catching a container-kill); flag low-severity CPU-micro *conservatively or not at all* (false positives there only erode trust).
- **Scope**: the diff √ó known hot paths. Don't review the world.
- **Confidence axis** on every finding: **flag-with-confidence** (mechanism-determined: allocation, dispatch, copying, crossing, unboundedness) vs **flag-as-measure** (opaque: depends on JIT/GC/optimizer decisions ‚Äî escape elision, inlining, GC impact).
- **Visibly-contestable perf tradeoff without data ‚Üí soft flag-as-measure.** Narrow trigger: the change makes a **visible tradeoff that could itself regress** ‚Äî removes a lock/guard/synchronization, swaps in a hand-rolled cache/structure, or explicitly claims "faster/optimized" ‚Äî **and** ships no benchmark/profile. Then a static read can't tell win from regression ‚Üí one soft flag-as-measure nudge ("trades X for Y ‚Äî verify with a JMH benchmark / JFR"). If nothing in the diff could plausibly regress, there's nothing to measure ‚Üí stay silent. Explicitly NOT fired for: a mechanically-obvious win (hoist-invariant, denser structure, removed allocation); routine adoption of a known-better idiom (lower-overhead builder/API/toolkit primitive, no visible downside); or a change that ships a benchmark/JFR (well-evidenced). One line, a nudge not a code-pattern finding.

## Hot-path map (where cost matters ‚Äî and the multiplier)
Span lifecycle (create / setTag / finish), tag map ops, serialization/encoding, the metrics/stats path, decorators, propagation (header read/write). **Multiplier: per-span √ó spans/request √ó requests/sec.** A per-span cost is multiplied massively; a per-process/once cost is negligible. The AI must reason about *which multiplier applies* before flagging.

## Universal checks (language-agnostic)
Format: **pattern** ‚Äî *expensive when (the interprocedural condition to trace)* ‚Äî confidence ‚Äî severity ‚Äî fix.

1. **Per-span/per-call allocation on a hot path** ‚Äî *per-span (or hotter) AND the object isn't trivially short-lived (it's retained, returned, captured, or passed across a boundary)* ‚Äî flag-with-confidence if clearly per-span + retained; flag-as-measure if lifetime/escape is borderline ‚Äî SEV-2/3 (‚ÜíSEV-1 if unbounded) ‚Äî fix: reuse, pool, dense/positional storage, defer out of the hot path.
2. **Repeat work across calls/traces** ‚Äî *string concat / case-conversion / regex *compile* / format / parse recomputed each hot-path call, on a recurring (low-cardinality) input or allocating each time* ‚Äî flag-with-confidence ‚Äî SEV-2/3 ‚Äî fix: memoize (bounded ‚Äî see #3) or compile-once (hoist regex to static).
3. **Unbounded memory / collection** ‚Äî *a cache/map/collection with no size+byte bound, or keyed by a high-cardinality input (per-request data, raw strings)* ‚Äî flag-with-confidence (unboundedness is structurally visible) ‚Äî **SEV-1 (OOM / container-kill ‚Äî top severity)** ‚Äî fix: bound by count *and* bytes; or don't cache high-cardinality inputs (opt out).
4. **Expensive work on the critical path that could be deferred** ‚Äî *heavy compute / parse / normalize / serialize / I/O / lock on the synchronous request or span-finish path, that could be moved* ‚Äî flag-as-*consider* (deferability is contextual) ‚Äî SEV-1/2 ‚Äî fix: defer to background/writer thread, lazy-compute, batch.
5. **Polymorphic dispatch on a hot path** ‚Äî *a hot call site becomes polymorphic enough to defeat the runtime's inlining/devirtualization (real for JIT runtimes ‚Äî JVM/.NET/V8; AOT/interpreted differ)* ‚Äî **flag-as-measure** ("may defeat devirtualization; verify on the target runtime") ‚Äî SEV-2/3 ‚Äî fix: keep hot call sites mono/bi-morphic; specialize.
6. **FFI / native-boundary crossing on a hot path** *(central to the shared-core effort)* ‚Äî *a native crossing per-span/per-item (not batched), or transporting strings/objects rather than primitives/IDs* ‚Äî flag-with-confidence (boundary cost is mechanism-determined; runtime-specific pinning ‚Üí addendum) ‚Äî SEV-1/2 (SEV-1 if it blocks/pins under concurrency) ‚Äî fix: batch (one per flush, not per item); transport interned IDs not strings; keep crossings off the hot/concurrency path.
7. **Escape / allocation-elision defeated** *(Java/Go/.NET/V8 all have a version)* ‚Äî *a refactor makes a previously-local object escape (stored, returned, captured by a closure, passed to a virtual/non-inlined call) ‚Üí silent heap allocation on a hot path* ‚Äî **flag-as-measure** ("may now escape and allocate; verify with an allocation profiler") ‚Äî SEV-2/3 ‚Äî fix: keep it local; avoid the escaping store/capture.

## Deterministic-lint candidates (DON'T spend AI budget ‚Äî make these real lints)
Fixed-signature, mechanically checkable:
- per-call cache / regex / expensive-object creation that should be static/once
- boxing in specific hot APIs
- using a string-API where an id-API exists on a hot decorator
- the existing convention rules (e.g. don't extract one-shot instrumentation methods to constants)
- *(grows as patterns prove mechanically checkable ‚Äî migrate them off the AI as they stabilize)*

## Java addendum (JVM-specific ‚Äî mechanism authored with JIT-developer authority; **calibrate production-priority against your own escalation history**)
Refines the universal checks with JVM mechanics. Quarantined here, for the Java audience that has the substrate.

**Scope (2026-07-08):** the primary optimization target is **C2 / Java 11+ (HotSpot)** ‚Äî the mechanisms below are stated in those terms (inline-cache/`TypeProfileWidth` model, C2 speculative inlining, EA). C1-only, OpenJ9/J9, GraalVM, and Java 8 should still benefit but are the minority case we don't *tune* for. Pairs with the benchmark JVM standardization (Java 17 HotSpot).

- **J1 ‚Äî Escaping allocation defeats Escape Analysis** *(refines #1, #7)*. The JVM scalar-replaces only *non-escaping* short-lived objects. An object stored in the tag map / span / a collection, iterated at serialization, or passed to a virtual/megamorphic call **escapes** ‚Üí EA can't elide it ‚Üí real heap allocation. The trap: *"the JIT will scalar-replace it" is false for escaping objects* ‚Äî the dense-store ‚àí48% came from removing exactly the escaping per-tag wrappers; an earlier no-Entry change measured ~0 because *those* entries were transient/EA-eligible. **Verify in JFR ‚Äî EA'd objects don't appear in alloc profiles, so a surviving `Entry` in the profile *proves* it escapes.**
- **J2 ‚Äî Megamorphic dispatch** *(refines #5)*. **Status ‚Äî PARKED for PR-review flagging (2026-07-08): do NOT raise megamorphism findings in review yet.** Kept as author-reference + the standing-audit target described below, not as an active review idiom. Rationale: it's too in-the-weeds to land with most devs, and the rubric must first bank *legible* wins (allocation, unbounded memory, regex ‚Äî what anyone can see in a profiler) to earn trust before deploying the subtle JIT checks. Revisit once the rubric has a track record. (Mechanism below stands; it's the flagging that's held.) A hot call site seeing **‚â•3 receiver types with no dominant one** goes megamorphic. **Do not frame this as "the virtual call is slow" ‚Äî a well-predicted indirect branch is a couple of cycles; the dispatch is a rounding error.** The cost is the **optimization fence the un-inlinable call erects on *both* sides**: caller-side, the args escape into an opaque callee ‚Üí no scalar-replacement/EA on them, no constant-propagation *into* the call, caller-saved registers spilled across it, no hoist/reorder across the boundary; callee-side, it's never specialized to *this* caller, so the arg types/constants that would have collapsed its internal branches and devirtualized its *own* downstream calls stay invisible. Inlining is what lets the two bodies optimize as **one unit**; the megamorphic call severs that ‚Äî on the hottest paths that is the whole bill. **C2 rescue conditions (the primary target):** ‚â§2 types stays **bimorphic** (inlinable); a **dominant receiver** (‚â•`TypeProfileMajorReceiverPercent`, default 90%) still gets guarded mono-inline + uncommon-trap fallback, so a *skewed* site is usually fine ‚Äî but watch **bimodal oscillation** (a recurring rare type ‚Üí deopt thrash). The danger zone is the **flat ‚â•3 distribution** (`TypeProfileWidth`=2 ‚Üí profile overflow ‚Üí itable v-call, no inline). Fixes, cheapest first: keep the site to ‚â§2 types; **gate-when-empty** so the common case skips the fan-out (a default-empty listener array); **collapse N impls into one** `final` class with an internal state/size-class switch (e.g. retire a legacy map impl so the optimized one is the sole impl ‚Üí every `TagMap` site monomorphic); or **call-site-split/unroll** a stable-order fan-out (catalog #7). **flag-as-measure** ‚Äî opaque; whether it bites depends on which impls actually load + the runtime receiver mix. Confirm with `-XX:+PrintInlining` (`not inlined (megamorphic)`) / JITWatch, not a code read. **Diff-review blind spot ‚Üí needs a standing audit.** The worst megamorphic sites *accumulate*: no single PR introduces them (each only nudges the type count by one), so a per-PR check catches only a PR that *widens* a site ‚Äî the pre-existing hazards are invisible to it. The heaviest is `Context.get`/`with` dispatching across `Empty`/`Singleton`/`Indexed`(+wrapper) impls ‚Äî the hottest path in the system (catalog #11). These want a **periodic PrintInlining census of the known hot sites**, run independent of any PR, not diff review.
- **J3 ‚Äî JNI / native crossing: overhead + virtual-thread pinning** *(refines #6)*. JNI call ‚âà 100ns‚Äì1¬µs (state transition, arg pin/copy, no inlining across); string args via `GetStringUTFChars` = UTF-16‚ÜíUTF-8 copy. **A JNI call from a virtual thread pins the carrier** ‚Üí no other vthreads on that carrier run while pinned ‚Üí concurrency collapse for vthread-reliant apps. Fix: batch at flush on the **writer (platform) thread**, keep the app-vthread path pure-Java, transport interned IDs not strings; `@CriticalNative` only for short primitive ops (no `JNIEnv`, no object args, no GC-safe state ‚Äî severe constraints). Flag the crossing + pinning risk (mechanism); overhead magnitude ‚Üí measure.
- **J4 ‚Äî GC pressure ‚Üí *tail* latency** *(refines #1)*. Hot-path allocation ‚Üí more GC ‚Üí STW pauses ‚Üí app **tail** latency, not just throughput (the tracer shares the app heap). ZGC has short pauses but isn't common ‚Äî assume G1/Parallel. flag-as-measure ("may raise tail latency; verify under load at a realistic heap").
- **J6 ‚Äî Reference strengthening in weak-cache scans** *(refines #1, #2)*. Calling `WeakReference.get()` (or `SoftReference.get()`) inside a cache-probe loop to identify the referent **strengthens** the reference ‚Äî the returned strong ref keeps the object alive until it goes out of scope, defeating the purpose of the weak reference. Pattern to flag: a loop over a weak-ref cache that calls `.get()` for identity/equality comparison on every slot probed. Fix: store a stable key (e.g. `System.identityHashCode(context)`) in the wrapper at construction time; compare the key first (plain int, no strengthening); call `.get()` only on a key match (the right moment ‚Äî you're about to use the referent anyway) or to detect eviction (`get() == null`). **flag-with-confidence** when `.get()` appears inside a probe loop on a hot path ‚Äî SEV-2/3.

- **J7 ‚Äî `substring`/slice ‚Üí `SubSequence` zero-copy view** *(refines #1, #7)*. `String.substring`/`subSequence` allocates a fresh backing array per call. On a hot parse path (headers, tags, query strings, SQL/DBM, propagation) where the slice is **transient** ‚Äî compared (`equals`/`startsWith`/`contains`/`indexOf`), parsed, or appended, then discarded ‚Äî a `SubSequence` (offset+length view) is zero-copy and EA-elided **iff** the consumer takes a `CharSequence`/range (else the boundary `toString()` erases the win ‚Üí add the overload or skip). **flag-as-measure** for the transient case (EA-dependent). The retention trap is **flag-with-confidence**: a `SubSequence` stored in a field/tag/collection/cache pins its *entire* backing String ‚Äî a small window over a large string is a net memory loss ‚Äî so a retained view must be materialized or `compact()`'d. Discriminator = transient (view, measure) vs retained (must detach).
- **J8 ‚Äî Backtracking regex on external input ‚Üí RE2J / bounded input** *(distinct from #2 compile-per-call)*. `java.util.regex` backtracks ‚Üí exponential worst-case (ReDoS) on adversarial input ‚Äî a CPU / tail-latency / DoS hazard, **not** an allocation one. Flag the **conjunction**: (a) input is user/external-controllable AND (b) the pattern is backtracking-prone (nested/overlapping quantifiers, `(a+)+`, unanchored `.*` around a quantified group). **flag-with-confidence** when both hold ‚Äî SEV-2 (tail latency), **SEV-1** on a per-request AppSec/security-scan path (IAST Reporter / WAF run regex on untrusted input every request). Fix: RE2J (`com.google.re2j`, guaranteed linear; no backrefs/lookaround), anchor/de-nest, or hard-cap input length.
- **J9 ‚Äî `Objects.hash(...)` varargs / boxing hash on a hot path ‚Üí `HashingUtils`** *(refines #1)*. `Objects.hash(a, b, ‚Ä¶)` is varargs ‚Äî it allocates an `Object[]` per call and **boxes every primitive** arg; `Arrays.hashCode` and hand-rolled `31*h + ‚Ä¶` combines that box also qualify. Per-span tag/key building, or the `hashCode()` of a hot value object, ‚Üí a guaranteed per-call allocation + boxing. Fix: `datadog.trace.util.HashingUtils` ‚Äî primitive `hash(long/int/boolean/char/‚Ä¶)` overloads (no boxing), `hash(Object,Object)` and `hash(int,int)` combiners (no array); for >2 fields fold pairwise through `hash(int,int)` (there is no varargs form, by design). flag-with-confidence (array + boxing are mechanism-determined) ‚Äî SEV-2/3.
- **J10 ‚Äî hot-path `String.format` / string munging ‚Üí `Strings` (+ `SubSequence`)** *(refines #2)*. `String.format` parses the format string, boxes its args, and allocates on every call ‚Äî never on a hot path; hand-rolled case-conversion, class/resource-name munging, blank-checks, and truncation recomputed per call qualify too. Fix: `datadog.trace.util.Strings` ‚Äî allocation-aware `replace`/`truncate(CharSequence)`/`isBlank`/`getResourceName`/`getClassName`/‚Ä¶; for **transient substring compares** prefer a `SubSequence` view (J7); for plain assembly, direct concatenation beats `format`. flag-with-confidence for `String.format` on a hot path; flag-as-measure for borderline munging ‚Äî SEV-2/3.
- **J11 ‚Äî composite / multi-dimensional key maps on a hot path ‚Üí `Hashtable` / `ConcurrentHashtable`** *(refines #1, #3)*. `Map<A, Map<B,V>>` nesting, or a `HashMap` keyed by a composite key (client-side stats, per-`(service, operation, ‚Ä¶)` aggregation), allocates nested maps + `Entry` objects + boxes keys on the hot aggregation path. Fix: `datadog.trace.util.Hashtable` (single-threaded, composite-key D1/D2 tables ‚Äî landed) or `datadog.trace.util.ConcurrentHashtable` (lock-free concurrent, **coming**) ‚Äî positional composite keys, fewer allocations. flag-as-measure ‚Äî SEV-2/3.
- **Toolkit availability ‚Äî cite only what exists.** Available today: `Strings`, `SubSequence`, `HashingUtils`, `Hashtable` (all `datadog.trace.util`), `RE2J` (`com.google.re2j`). Coming (name as "coming", don't imply it's present): `ConcurrentHashtable`, `StringIndex` (immutable string set/map), `UTF8BytesString.Cache` (recurring-string interner), wider `IntegerCache` (http-status/port boxing), `DDCache` inlining. **J7‚ÄìJ11 route an *existing* #1/#2/#3 finding to a reusable fix ‚Äî they are not new flag-triggers. Don't raise a finding you wouldn't have raised anyway; the posture (precision, silent-when-unsure, findings-cap-scales-with-diff ‚Äî see `SKILL.md`) is unchanged.**
- **J5 ‚Äî Cardinality-sensitive aggregator** *(domain-specialized #3)*. Some structures are invisible to the generic "unbounded collection" check because the risk is *cardinality*, not raw size: a config- or user-driven value (tag key, resource name, HTTP URL) feeding a **cardinality-sensitive aggregator** (e.g. the conflating metrics aggregator ‚Äî each unique label combination = one aggregate; a `maxAggregates` cap bounds OOM but high-cardinality input *thrashes* it: constant eviction, garbled metrics). flag-with-confidence when config/user-driven values reach an aggregator with a per-key budget ‚Äî **SEV-1** (same class as unbounded memory: correctness + heap impact). Fix: bound the source cardinality before it enters the aggregator, or use sentinel substitution for over-cap values. This surfaced on merged production code more than once in back-test calibration ‚Äî the capstone pattern where the bot's value concentrates.

## Instrumentation (ByteBuddy Advice) idioms ‚Äî dd-trace-java-specific fixes
The core rule is a **predicate-with-default, not a banned-API list**: don't flag "you called `String.format`"; flag *"an eager, unconditional expensive call on an instrumentation-reachable path."* Two failure shapes, different fixes: (1) the result is usually **discarded** ‚Üí **gate/defer** (parameterized logging, `isEnabled()` guard) ‚Äî the cost is avoidable; (2) the result is always **needed but costly** ‚Üí **cheapen/cache** ‚Äî gating does nothing. The discriminator (eager? unconditional? guarded? result-needed?) is interprocedural ‚Äî that's the review's job (¬ß workflow). These idioms are the actionable fixes when a universal/Java finding lands on an `@Advice` path:

- **`Config.get()` / `InstrumenterConfig.get()` on a hot path ‚Äî flag-with-confidence.** Walks a config-resolution chain; not a free read. Fix: hoist to a `static final` field, or compute once in the constructor. Common trap: the call is buried in a helper invisible at the advice site ‚Äî grep transitively. (The single most recurring finding in calibration ‚Äî five independent occurrences.)
- **`@Advice.AllArguments()` ‚Äî deterministic lint.** Materializes a new `Object[]` boxing all arguments on every advised call; always escapes, EA cannot elide it. Fix: `@Advice.Argument(value=N)` for the one argument and type needed.
- **`@Advice.SkipOn(OnDefaultValue.class)` + cached boolean ‚Äî the preferred feature-flag pattern.** Compute a `static final boolean` once, return it from `@Advice.OnMethodEnter`, suppress exit advice when disabled. Residual cost: one JIT-hoistable field read per call. Recommend this whenever a `Config.get()` shows up in advice.
- **`@Advice.Local` ‚Äî prefer over `ThreadLocal`.** Carries per-invocation state from `OnMethodEnter` to `OnMethodExit` with no map lookup.
- **`@Advice.Origin` String allocation.** Injects class/method name as a `String` parameter ‚Äî allocates per invocation at a non-static site. Prefer a `static final String` constant.
- **Java Stream API on an advice/hot path ‚Äî flag-with-confidence.** `stream()`/`IntStream` allocate `Spliterator` + pipeline objects per call; JIT elision is fragile. Fix: a plain `for` loop (`cstyleFor`/`enhancedFor`/`forEach`/`iterator` are all on par, ‚âà0 alloc). See guide ¬ß2 for the benchmark numbers and the three silencing exceptions (cache-miss body, length-guarded error path, cold/startup).
- **`switch(String)` ‚Äî three-tier fix ranking.** (1) resolve to a constant `long` id (folds on any JIT); (2) open-addressed table (never folds but always inlinable, low-variance); (3) plain `switch` (fine for small, inlinable dispatch). Flag *large* switches (inline-budget exhaustion is mechanism-certain); flag *small* switches on hot constant-arg paths only as a version-conditional soft-alert.
